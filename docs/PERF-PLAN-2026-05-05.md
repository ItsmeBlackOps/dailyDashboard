## Section A — DBA findings (measured, not guessed)

### 🔥 A1. `taskBody.Recruiter` / `taskBody.Expert` / `taskBody.Branch` are NOT indexed

The most impactful finding. `_scopeFilter` runs on every candidate-list load and uses:
```js
{ $or: [{ Recruiter: { $in: regexes } }, { Expert: { $in: regexes } }] }
// or  { Branch: regex }   for mm/manager
```

Explain plan against prod **right now**:
```
totalDocsExamined: 17,933 (FULL COLLECTION SCAN)
nReturned: 0
executionTimeMillis: 35
```

Every dashboard load by every manager/AM/lead does this. With concurrent users the 35ms compounds.

**Fix:** create indexes. Non-destructive, builds in background.
```js
db.taskBody.createIndex({ Recruiter: 1 }, { name: "Recruiter_1" });
db.taskBody.createIndex({ Expert: 1 },    { name: "Expert_1" });
db.taskBody.createIndex({ Branch: 1 },    { name: "Branch_1" });
```

**Expected after:** `totalDocsExamined` drops to scan size of matched index entries (typically <100), millis ≈ 1-3.

### 🔥 A2. `taskBody` has 19 indexes — duplicates are 5x writes/insert

Three near-identical subject indexes:
- `subject_1` — `{subject: 1}` (lowercase)
- `task_subject_ci_idx` — `{subject: 1}` (lowercase)  ← exact dup
- `task_Subject_ci_idx` — `{Subject: 1}` (capital)

Also redundant single-field indexes covered by composites:
- `receivedDateTime_-1` covered by `status_1_receivedDateTime_-1` for status-leading queries
- `assignedTo_1` covered by `assignedTo_1_receivedDateTime_-1` for assignedTo-leading queries

**Fix:** drop the duplicates. 18,000 inserts/year × 3 fewer index writes each = real WAL/IO savings; less storage; less cache pollution.
```js
db.taskBody.dropIndex("task_subject_ci_idx");        // exact dup of subject_1
db.taskBody.dropIndex("assignedTo_1");               // covered by composite
db.taskBody.dropIndex("receivedDateTime_-1");        // covered by composites
// Verify no query plans depend on these via $indexStats before dropping.
```

### A3. `auditLog` has no `phase` index — Processing Logs tab gets slower with every audit row

72,358 docs and growing. Admin queries hit `{phase: 'AUTO_ASSIGN_FAILED', timestamp: range}`. Current indexes are `{subject, timestamp}` and `{timestamp}` only — neither helps the phase filter, so Mongo does a date-range index scan then in-memory filter.

**Fix:**
```js
db.auditLog.createIndex(
  { phase: 1, timestamp: -1 },
  { name: "phase_1_timestamp_-1" }
);
```

### A4. Case-insensitive regex on indexed fields = silent COLLSCAN

`buildDashboardRoleMatch` in `models/Task.js` uses `{cc: { $regex: 'tushar.ahuja', $options: 'i' }}`. Even though `cc` IS indexed, the `$options: 'i'` flag forces a scan because the index has no collation.

**Fix (two options):**

- **(a) Maintain lowercased duplicate fields** on write: `assignedTo_lc`, `cc_lc`, `sender_lc`. Indexed. Query becomes exact match. **Best long-term.**
- **(b) Add case-insensitive collation indexes** on these fields. Easier to ship but compounds storage.

Recommendation: **(a)**, ship as a backfill + ingestion-path PR. ~150 LOC.

### A5. `candidateDetails.Branch` missing index

mm/manager scope filter on candidates falls into `_scopeFilter`'s Branch branch — `Branch: regex`. `candidateDetails` already has Recruiter + Expert indexes but is missing Branch.

```js
db.candidateDetails.createIndex({ Branch: 1 }, { name: "Branch_1" });
```

## Section B — Backend findings

### B1. `taskService.getDashboardSummary` already parallelized ✅
PR #109 — done. Wall-clock halved.

### B2. `taskModel.getDashboardSummary` aggregation has implicit `$last` without `$sort`

```js
{ $group: { _id: "$Candidate Name", rounds: { $push: "$normalizedRound" },
            lastDocument: { $last: "$$ROOT" } } }
```

`$last` is non-deterministic without a preceding `$sort`. Also pushes `$$ROOT` (the whole doc) into the group — memory waste.

**Fix:** add `$sort: { receivedDateTime: -1 }` before `$group`, and project only the lastDocument fields you actually use instead of `$$ROOT`.

### B3. `/api/admin/interview-support/logs` reads up to 2000 rows then filters in JS

After PR #123 the endpoint queries the full day's auditLog (2,300 today). It groups + paginates in memory.

**Fix:** add server-side pagination + the A3 index. Frontend takes a `page` and `limit` param.

### B4. Frontend makes 4-8 sequential `authFetch` on initial dashboard load

TasksToday, BranchCandidates, ProfileHub all fire multiple endpoints serially in `useEffect`. Each costs network RTT + backend response time.

**Fix:** combine the 4-8 calls into a single `/api/dashboard/bootstrap?context=tasks-today` that fans out server-side with Promise.all and returns one payload. Or use HTTP/2 server push if your stack supports it (Vercel + Vite does).

### B5. Every page load reissues `/api/users/manageable` even though the list barely changes

The role list almost never changes within a session. React Query with `staleTime: 5 minutes` would eliminate ~90% of these calls.

## Section C — Frontend findings

### C1. No code splitting on the auth-gated routes

`App.tsx` eager-imports `TasksToday`, `Index` (DashboardV2 wrapper), `UserManagement`, `PermissionsManagement`. Combined those are 200-300KB minified. Every login pays the cost even if the user never visits some.

**Fix:** convert the eager imports to lazy:
```tsx
const TasksToday = lazy(() => import('./pages/TasksToday'));
const UserManagementPage = lazy(() => import('./pages/UserManagement'));
```
Already lazy: Reports, ReportAssistant, JobsPage, BranchCandidatesPage, etc. Just standardize.

### C2. TasksToday is 4000+ LOC in one file

Loaded on every login. Long parse + initial render. Breaking it into:
- `TasksToday.tsx` (container, ~500 LOC)
- `TaskTable.tsx`, `TaskFilters.tsx`, `MeetingPanel.tsx`, etc.
... wouldn't reduce bundle but does help React tree-shaking and parallel parsing of TSX with Vite's worker.

### C3. No prefetching on sidebar hover

When user hovers "Reports" we know they're about to navigate. Prefetch the chunk + the data:
```tsx
const prefetch = () => { import('@/pages/Reports'); };
<NavItem onMouseEnter={prefetch} ... />
```
~50ms perceived speedup per nav.

### C4. React Query not used everywhere

Some pages use `useQuery` (admin tabs do), others use raw `useEffect + fetch`. Consistency = automatic caching, deduplication, background refetch. Migrating the holdouts is mostly mechanical.

Specific places to migrate (these refetch on every mount):
- `TasksToday.tsx` — `/api/users/manageable`, `/api/users/active` (PR #104 added shimming but uses bare `useEffect`)
- `UserManagement.tsx` — `/api/users/manageable`
- `BranchCandidates.tsx` — `getBranchCandidates` socket (caching socket events is harder but possible via `useSyncExternalStore`)

### C5. Bundle analyzer hasn't been run

Vite has `rollup-plugin-visualizer`. Run once, identify the heaviest dependency (likely Radix UI + Lucide icons + moment-timezone), tree-shake or replace:
- **moment-timezone** is 250KB+. Replace with `Intl.DateTimeFormat` for display + `date-fns-tz` for math. ~150KB drop.
- **Lucide** — all icons bundled if you `import { Foo } from 'lucide-react'`. Switch to per-icon imports.

### C6. PostHog initialization + New Relic + MSAL all run at app mount

These cost 100-300ms of main-thread JS before first paint. Defer to after first interaction:
```tsx
useEffect(() => {
  requestIdleCallback(() => { posthog.init(...); newrelic.start(...); });
}, []);
```

## Section E — Branch Candidates audit (added 2026-05-05)

The page that lights up the most user complaints. Audited line-by-line. Five distinct perf issues:

### E1. 4000-LOC single file rendering 100+ table rows = i5 5th gen pain
`components/dashboard/BranchCandidates.tsx` is one component containing the page, the table, the create dialog, the edit drawer, the support sheets (assessment/interview/mock), and the resume understanding flow. On the user's hardware (4-8 GB RAM, i5 5th gen) the initial parse + first render is the biggest single source of "page is slow".

**Fix:** lazy-import the per-row drawers and support sheets:
```tsx
const PODraftSheet = lazy(() => import('@/components/shared/PODraftSheet'));
const SupportAssessmentSheet = lazy(() => import('./SupportAssessmentSheet'));
const SupportInterviewSheet = lazy(() => import('./SupportInterviewSheet'));
```
These are opened on click — no reason to ship them in the main chunk.

### E2. No table virtualization
Every candidate row is rendered to the DOM. For a manager seeing 200+ candidates that's 200+ React subtrees + 200+ status badges + 200+ avatars. On i5 5th gen this is the single biggest UI freeze.

**Fix:** `@tanstack/react-virtual` over the candidate `TableBody`. Drops DOM nodes from 200 to ~30 (viewport-sized). 4-5× scroll perf, 2× initial render speed. ~50 LOC change.

### E3. End-clients dropdown was slow on every new-record dialog open
Was: `distinct('End Client')` over scoped candidateDetails + `find()` on endClients curated list — every CompanyCombobox mount.
**Now fixed (this PR):** 5min in-memory cache per user, invalidated on add.

### E4. Status-update emit fires for one row then re-fetches the whole list
`updateCandidateStatus` socket call triggers a full `getBranchCandidates` re-fetch instead of patching the single row in state. On a manager view this re-runs the BFS + delegation union + scope filter.

**Fix:** the socket emit returns the updated row already. Patch state in place:
```ts
socket.emit('updateCandidateStatus', { ... }, (resp) => {
  setCandidates(prev => prev.map(c => c.id === id ? { ...c, status: resp.candidate.status } : c));
});
```
No backend change. Eliminates a 1-2s round trip on every status click.

### E5. Backend BFS is unbounded for managers
`fetchAllCandidates` returns every candidate matching scope with no pagination. Once you hit 500+ candidates the JSON serialization alone is 500KB+.

**Fix:** server-side cursor with `limit: 200, skip: page * 200`. Frontend adds prev/next. Or infinite-scroll with the virtualization from E2.

## Section F — Dashboard audit

### F1. KpiOverview loads ALL tasks for the day
`taskService.getDashboardSummary` reads every task in the date range (could be 500+ for a manager) and computes KPIs in JS. Already parallelized (PR #109) but still O(n).

**Fix:** pre-aggregate on the backend with `$facet`:
```js
{ $facet: {
  totals: [{ $count: 'count' }],
  byStatus: [{ $group: { _id: '$status', count: { $sum: 1 } } }],
  byActualRound: [{ $group: { _id: '$Actual Round', count: { $sum: 1 } } }],
}}
```
Returns 3 numbers + 2 small breakdowns instead of 500 rows.

### F2. Stats endpoint hits Mongo 4× sequentially
`getStats` does 4 sequential `await`s on the same collection. Combine into one aggregation with `$facet`. Same shape as F1.

### F3. DashboardV2 tabs all mount even when hidden
Each `TabsContent` renders its tree even when not active in Radix's default mode. RecruiterAnalytics and ExpertAnalytics both fire fetches in `useEffect` on mount.

**Fix:** wrap each tab's body in `<Suspense>` and lazy-import the component. Tab activation triggers the import + fetch. Defer everything not on the "Overview" landing tab.

## Section G — Low-end hardware (4-8 GB RAM, i5 5th gen) considerations

All of the following matter MORE on this hardware than they would on a developer's M-series MacBook:

| Concern | Why it bites low-end | Mitigation |
|---|---|---|
| Bundle size | Slower JS parse/compile on older CPUs | Aggressive code splitting + tree-shaking + moment-timezone → date-fns-tz |
| DOM node count | DOM layout is single-threaded; older CPUs choke at >500 rows | Virtualize tables (E2) + paginate (E5) |
| Re-renders | Each re-render = full diff + paint; missing useMemo bites hard | Profile with React DevTools, add memo where lists render |
| In-flight requests | Every concurrent fetch keeps a connection + decoder running | Coalesce via React Query, dedupe at the boundary |
| Animations on hover/transition | Compositor frames drop under JS pressure | Replace heavy framer-motion animations with CSS transitions on hot paths |
| Image / SVG icons | All Lucide imports bundles every icon | Per-icon imports — Vite tree-shakes |

**Specific items to add/promote in the plan:**
- **Promote C5 (bundle analyzer + moment-timezone swap) to Day 2** — biggest single bundle win.
- **Promote E2 (virtualization) to Day 3** — biggest single render-time win.
- **Demote stretch items that increase bundle size** — don't add new dependencies unless they replace something larger.
