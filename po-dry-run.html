Single point of write — setCandidateStatus() service helper

Every status change goes through ONE function. Direct $set: {status: ...} writes are blocked at the model layer. The helper does both updates atomically:

// backend/src/services/candidateStatusService.js
async function setCandidateStatus({ candidateId, newStatus, ctx }) {
  // ctx = { source, changedBy, reason?, sourceRef?, sideEffects? }
  const before = await col.findOne({ _id }, { projection: { status: 1 } });
  if (before?.status === newStatus) return { skipped: true, reason: 'no-op' };

  const entry = {
    from:      before?.status ?? null,
    to:        newStatus,
    changedAt: new Date(),
    changedBy: ctx.changedBy,
    reason:    ctx.reason ?? null,
    source:    ctx.source,         // 'po-email' | 'manual-ui' | ...
    sourceRef: ctx.sourceRef ?? null,
  };

  const setFields = { status: newStatus };
  if (newStatus === 'Placement Offer' && ctx.poDate) setFields.poDate = ctx.poDate;
  if (ctx.placement) setFields.placement = ctx.placement;

  await col.updateOne({ _id }, {
    $set: setFields,
    $push: { statusHistory: entry },
  });
  return { changed: true, entry };
}

Enforcement — change-stream watcher

Mongo change stream on candidateDetails.status. Every modification verifies a matching statusHistory entry was pushed within the same operation. If a stream event shows status changed but the corresponding history entry is missing, an alert fires (means someone bypassed the helper).

PO automation hooks into this

  • Each of the 54 SET emails would call setCandidateStatus() with source: 'po-email', sourceRef: { kind: 'outlookMessage', id: messageId }, reason: 'PO arrived' — history entry recorded automatically.
  • The 22 CORRECT emails update poDate only (status already PO) — still push a history entry with from === to === 'Placement Offer', reason: 'poDate corrected by N days'. This explains the date change in the audit trail.
  • The 13 SKIP emails write nothing (no-op short-circuits in the helper).
  • The 10 QUEUE emails write to poEmailIngestionLog with status: 'unmatched' — no candidateDetails touched.

Phase 2 — PO inference from email archive: for each of Ronak's 99 historical PO emails (Feb 9 - Apr 30), if the candidate exists, push a synthetic transition entry { from: 'Active', to: 'Placement Offer', changedAt: emailReceivedDate, source: 'po-email', sourceRef }.   3. Phase 3 — live mode: all new status changes flow through setCandidateStatus(). Block direct writes via Mongoose pre-save / model layer.
