Reading guide: each item below has 4 fixed sections — What it is · Where the code lives · Concrete example with real users · Does it change role-based access? Tags at the top of each card make the access-impact obvious at a glance.

C8 — Hierarchy-aware profile read

SCOPEDDOES NOT CHANGE WHO CAN MANAGE ANYONE
What it is Today, GET /api/users/profile/:email only lets you read your own profile OR have role admin/mm. C8 broadens that to also include leads/managers reading profiles within their own hierarchy. Read-only — does not grant any new mutation power.
Where the code lives
File · LineWhat it does today
backend/src/controllers/userController.js:139-142Hard 403 unless requester is admin or mm
backend/src/services/candidateService.js:228-275Existing collectHierarchyEmails() that already does the BFS — re-use this
backend/src/services/userService.jsAdd new helper isUserInRequesterHierarchy(requester, targetEmail)
Concrete example
Today (broken)
Umang Pandya (role: lead) clicks "View profile" on Vaibhav Kumar (one of his 5 direct reports).
→ GET /api/users/profile/vaibhav.kumar@silverspaceinc.com
→ Server: requester role is lead — not in ['admin','mm'] → 403 Forbidden.
Same goes for Tushar Ahuja (MM) trying to view any of his 77 recruiters — wait, Tushar IS mm, so he's fine. But Brhamdev Sharma (MAM) trying to view a recruiter under him? 403.
After C8
The check becomes "admin OR mm OR target is in requester's hierarchy". Umang queries Vaibhav → BFS via teamLead = "Umang Pandya" finds Vaibhav → 200 OK. Brhamdev viewing one of his recruiters → walks down through his MLeads → 200 OK. Anyone outside their tree still gets 403.
Does it change role-based access? No. This relaxes a read-only viewport, not management permissions. A lead still cannot promote, demote, delete, or reassign anyone. The only thing that changes is "403 on a profile-read API" → "200 when target is in your team."

C9 — Migrate teamLead from display-name string to email reference

LARGEST CHANGEDOES NOT CHANGE WHO CAN MANAGE ANYONESCHEMA + BACKFILL
What it is Currently users.teamLead stores a display name string like "Umang Pandya". The hierarchy BFS matches that string against profile.displayName derived from emails. Five different functions across the codebase do this string-matching dance, and any time profile.displayName drifts (typo, name change, marriage), the whole tree breaks silently. C9 changes teamLead to store the user's email — a stable foreign key.

frontend/src/pages/UserManagement.tsx:155-200, 532-680UI binds teamLead as a free-text formatted nameDatalist of active users; submitted value is email
One-shot mongosh script—Convert all 251 teamLead display-name strings to corresponding emails
Concrete example
Today (fragile)
Vaibhav Kumar's doc has teamLead: "Umang Pandya". The BFS to find all of Umang's reports does:
const leaderName = deriveDisplayNameFromEmail("umang.pandya@silverspaceinc.com")
                    // → "Umang Pandya"
const directs = leadToUsers.get(leaderName.toLowerCase())
If Umang's email ever changes to upandya@…, the derive function returns "Upandya" instead of "Umang Pandya". The map lookup fails. Vaibhav becomes orphan. Hierarchy silently disappears.
After C9
Vaibhav's doc has teamLead: "umang.pandya@silverspaceinc.com". BFS becomes:
const directs = await db.users.find({ teamLead: requester.email }).toArray()
Email change, name change, display-name typo — all irrelevant. teamLead = email of an active user, stable forever.
Does it change role-based access? No. Same hierarchy semantics — same person sees same set of subordinates. The difference is robustness — it stops silently breaking when display names drift. Fewer "why doesn't user X show up under user Y?" support tickets.

C12 — Lead can promote/manage another lead

EXPANDS RBAC SLIGHTLYPAIRED WITH C8
What it is Currently a lead can only create or manage role user. They can't promote a peer, can't demote a peer, can't bring in another lead. Every reorg requires Harsh Patel (admin). C12 adds "lead" to the set of roles a lead can create and manage within their own subtree.

Concrete example
Today
Bhavya Dutt (lead) wants to promote Sonali Das to lead-tier (Sonali has been mentoring two people). Bhavya cannot. Has to ping Harsh.
After C12
Bhavya goes to User Management → picks Sonali → Role dropdown now shows both "lead" and "user" → promotes. Audit log captures it. Harsh stays out of the loop.
Does it change role-based access? Yes — slightly. This is the one backlog item that does. Leads gain a new capability they don't have today: lateral peer promotion. Should land paired with C8 (so leads can also see peer profiles to know who to promote) and a hierarchy-scoped enforcement (so lead-A can't promote lead-B who's in lead-C's tree). The hierarchy scope is the safety belt.

C16 — Pre-save validator for teamLead/manager values (the "mechanism" from D3)

PREVENTIVEDOES NOT CHANGE WHO CAN MANAGE ANYONE
What it is The reason we even had issues like "Lead A", "Manager A", and "\"aman Sagar\" <aman Sagar" stuck in the database is that nothing at the write layer rejected them. C16 adds a model-level pre-save validator that rejects:
  • self-loop (user's own display name as their teamLead)
  • strings containing < > " or other Outlook artifacts
  • placeholder strings ("Lead A", "Manager A", "TBD", "???")
  • values that don't resolve to an active user (with C9, this is just "email exists in users with active=true")
Where the code lives
File · LineCurrent behaviorAfter C16
backend/src/models/User.js:184-260updateUser writes whatever you give itValidates teamLead + manager before $set; throws 422 on bad input
backend/src/services/userService.js—New validateTeamLeadReference(value) shared helper
backend/src/services/userService.js:625-680 (bulkUpdateUsers)Allows malformed refs throughRejects rows with bad refs and reports them in failures[]
Concrete example
Today
A buggy script (or a careless admin paste) sets users.updateOne({...}, {$set: {teamLead: '"Aman Sagar" <aman.sagar@…>'}}) — write succeeds. Hierarchy traversal breaks for that user silently. Took the audit (Section 5 of the previous report) to even discover it.
After C16
Same write hits the validator → 422 Invalid teamLead reference: contains forbidden characters <. The bad data never lands. The dashboard / API return a clear error so the operator fixes it on the spot.
Does it change role-based access? No. Validators only reject malformed data. Whoever could write a teamLead before C16 can still write a teamLead after C16 — they just can't put garbage in it.
